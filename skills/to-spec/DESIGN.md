# Title of the DESIGN

<!--
The DESIGN document explains how the solution will be built. It should translate the PRD's user-facing needs and the RFC's proposal into an implementable technical plan.

Begin with a short overview: what is being designed, what problem it solves, and the high-level shape of the solution. A reader should understand the intended implementation approach before reading the details.
-->

## Goals

<!--
List the concrete technical goals this design must satisfy. These should be specific enough that an implementer can make tradeoffs against them.
-->

## Non-Goals

<!--
List anything intentionally out of scope. Non-goals prevent the design from expanding beyond the agreed PRD/RFC boundaries.
-->

## Current State

<!--
Describe the relevant existing system, code paths, data structures, integrations, and constraints. Link to files, docs, prior RFCs, or decisions where useful.
-->

## Proposed Design

<!--
Describe the chosen technical approach. Cover the main components, responsibilities, boundaries, and how they interact. This is the core implementation plan.
-->

## User and System Flows

<!--
Describe the important flows step by step. Include both happy paths and meaningful edge cases. Sequence diagrams or pseudocode are useful if they clarify the design.
-->

## Data Model

<!--
Document any new or changed data structures, schemas, persisted fields, config, events, or API payloads. Explain ownership, validation, migration, and compatibility concerns.
-->

## Interfaces

<!--
Document public APIs, internal interfaces, commands, UI surfaces, extension points, or integration contracts introduced or changed by this design.
-->

## Error Handling and Edge Cases

<!--
Explain how the system behaves when inputs are invalid, dependencies fail, permissions are missing, state is stale, operations are retried, or partial completion occurs.
-->

## Security, Privacy, and Permissions

<!--
Describe security considerations, access control, sensitive data handling, privacy implications, and any threat-model concerns.
-->

## Observability

<!--
Describe logging, metrics, tracing, audit events, debugging tools, and operational signals needed to understand whether the feature works in production.
-->

## Testing Plan

<!--
Describe unit, integration, end-to-end, regression, fixture, and manual tests. Tie tests back to PRD user stories or RFC requirements where possible.
-->

## Rollout and Migration

<!--
Describe feature flags, staged rollout, backwards compatibility, migrations, fallback behavior, and rollback plans.
-->

## Alternatives Considered

<!--
Document serious alternatives and why they were not chosen. Preserve useful context so future readers do not repeat the same investigation.
-->

## Open Questions

<!--
List unresolved questions, decisions needed, owners, and follow-up tasks. Keep this section current as the design evolves.
-->
